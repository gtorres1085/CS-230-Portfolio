## Software Design Document Reflection

Client: The Gaming Room

Summary of the Client and Requirements
The client, The Gaming Room, wanted to expand their game Draw It or Lose It from Android to multiple platforms. The application is a multiplayer game where teams guess images rendered over time. The main requirements included supporting multiple users at once, running across different operating systems, and being scalable and reliable. The system also needed to handle game sessions, manage users and teams, and ensure smooth performance across platforms.

What I Did Well
One thing I did well was clearly organizing the design document and breaking down system requirements into smaller, understandable sections. I also did a good job explaining the architecture, especially the client-server model and how REST APIs allow communication between systems. This made the design easier to follow and showed how the system could scale.

How the Design Document Helped Development
Working through the design document helped me understand the system before writing any code. It forced me to think about structure, data flow, and how different parts of the application interact. This made development more efficient because I already had a clear plan instead of figuring things out as I coded.

What I Would Improve
If I could revise one part, I would improve the UML diagrams and add more detail. While they explained the structure, they could be clearer and more polished. I would also include more real-world examples of how components interact to make the design even easier to understand.

Understanding User Needs
I interpreted the user’s needs by focusing on performance, accessibility, and usability. Since this is a multiplayer game, it was important to design a system that could handle multiple users at once without lag. Considering user needs is critical because the software must solve real problems and provide a good experience, not just function correctly.

My Design Approach
I approached the design by breaking the system into smaller components and using a client-server architecture with RESTful communication. I also focused on scalability and platform independence by recommending a cloud-based solution. In the future, I would continue using strategies like modular design, UML diagrams, and early planning to build efficient and scalable applications.
